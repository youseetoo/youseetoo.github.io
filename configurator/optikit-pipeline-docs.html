openUC2 · technical reference · iteration 2

From cubes on a grid→
to a runnable optical system.

OptiKit lets a researcher snap optical modules onto a 50 mm baseplate. Optiland traces light through the resulting system. The optikit-exporter is the bridge: it reads a flat layout JSON, looks up each module in a CSV database, walks the beam from every detected light source until it hits a sensor, and emits a fully populated Optiland JSON that you can open and simulate. This page documents that pipeline at every step.

02 Stage 1 · Layout ingestion

The layout file is your hardware manifest.

A layout JSON lists each cube on the baseplate together with its grid position, rotation, and any per-instance overrides. The exporter parses it into typed LayoutElement records — nothing more.

What the file looks like

Each entry under uc2_components describes a single cube. Two fields drive everything else: moduleId (the database key) and grid_pos (integer cell coordinates).

optikit-layout-optikit.json{
  "name": "my-bench",
  "uc2_components": [
    {
      "name":        "Source_00",
      "moduleId":    "laser-635nm",        # key into the registry
      "grid_pos":    [0, 5, 0],            # integer cube indices on the 50 mm grid
      "rotation":    [0, 0, 0],            # Euler degrees; yaw drives propagation axis
      "params":      { "power": 10 }      # per-instance override (wins over CSV)
    },
    {
      "name":        "Lens_00",
      "moduleId":    "lens-pos-1x1",
      "grid_pos":    [3, 5, 0],
      "rotation":    [0, 0, 0],
      "params":      { "focalLength": 50 }
    }
  ]
}

How the parser models it

The loader hard-codes one constant — the 50 mm grid pitch — and converts integer cells into millimetre coordinates via a property:

layout_loader.pyGRID_PITCH_MM = 50.0

@dataclass
class LayoutElement:
    name:         str
    module_id:    str
    grid_pos:     tuple[int, int, int]
    rotation_deg: tuple[float, float, float]
    params:       dict[str, Any]

    @property
    def cube_center_mm(self):
        ix, iy, iz = self.grid_pos
        return (
            ix * GRID_PITCH_MM + GRID_PITCH_MM / 2,
            iy * GRID_PITCH_MM + GRID_PITCH_MM / 2,
            iz * GRID_PITCH_MM + GRID_PITCH_MM / 2,
        )

So a cube at grid_pos=[3, 5, 0] centres at (175, 275, 25) mm in world space. The rotation is kept as raw degrees — the beam router will interpret yaw as an axis later.

03 Stage 2 · Module registry

Three layered sources of truth.

The registry decides what every moduleId means optically. It loads in passes — each pass overrides the previous when both define the same module.

Fig. 03 · three layers · later layers override earlier ones for the same id

The factory call

optikit_to_optiland.py · export_layout()registry = ModuleRegistry.from_csv(
    csv_path     = modules_csv,        # defaults to data/modules_updated.csv
    modules_root = modules_root,       # optional ../openUC2-optikit-modules
)

Inside from_csv, the registry walks the CSV row-by-row, skips rows whose opticalKind column is empty (those are stages, controllers, brackets — non-optical), and stores a spec dict per moduleId. If a modules root is supplied, it then scans for .../optics/optiland.json files and replaces matching CSV entries.

What a spec dict looks like

The same shape every consumer expects, regardless of where the data came from:

spec dict (registry.resolve return value){
    "kind":       "ideal",                 # ideal | optiland_json | zemax | vendor_catalog
    "ideal_kind": "thin_lens",             # sub-type for the builder dispatcher
    "params": {
        "focalLength": 50.0,             # merged: instance > CSV row > defaults
        "diameter":    25.4,
        "material":    "N-BK7",
        "dx_mm":       0.0,             # intra-cube placement (always present)
        "dy_mm":       0.0,
        "dz_mm":       0.0,
        "rxOffset_deg": 0.0,
        "ryOffset_deg": 0.0,
        "rzOffset_deg": 0.0,
    },
    "module_id": "lens-pos-1x1",
}

Resolution rules

  1. If nothing is registered — the moduleId becomes {kind: ideal, ideal_kind: passive}. The exporter never crashes on an unknown id; it just emits a transparent waypoint.
  2. If the CSV row has opticalModel set — kind is bumped to optiland_json (or zemax for .zmx extensions) and source points to the file.
  3. If the modules root supplies a real optiland.json — the spec is rewritten to {kind: optiland_json, source: …} while keeping the CSV-derived ideal_kind and params.
  4. Per-instance params from the layout — merged on top of the spec at resolve() call time. Instance always wins.
Sources are detected here. A "source" is just any module whose resolved ideal_kind == "source". The beam router asks the registry one question per element — _is_source, _is_mirror, _is_detector — and decides the chain accordingly.

How a row becomes a spec

The translation lives in module_registry.py · _row_to_spec(). Numeric cells are coerced through _to_float; non-numeric cells survive as strings; placement keys default to 0.0 so the orchestrator can rely on them unconditionally:

module_registry.py · _row_to_spec()def _row_to_spec(row):
    params = {}
    for column, dst in _COLUMN_TO_PARAM:        # 38 column → param name pairs
        raw = row.get(column)
        if raw in (None, ""): continue
        num = _to_float(raw)
        params[dst] = num if num is not None else raw
    for key in _PLACEMENT_KEYS:                   # dx_mm, dy_mm, dz_mm, rxOffset_deg, ...
        params[key] = _to_float(row.get(key)) or 0.0
    return {
        "kind":       (row.get("opticalKind") or "ideal").strip(),
        "ideal_kind": (row.get("idealKind") or "passive").strip() or "passive",
        "params":     params,
        "module_id":  row["id"],
    }
05 Stage 3 · Beam routing

Walking the grid, one cell at a time.

The layout doesn't encode beam order; the router reconstructs it. Start at every detected source, step one grid cell forward, redirect at every mirror, stop at the first detector.

How a yaw becomes a propagation axis

A source's yaw (the second number in rotation) selects which axis the beam travels along:

beam_router.py_YAW_TO_AXIS = {
    0:   "+x",
    90:  "+y",
    180: "-x",
    270: "-y",
}

From there, every step on the grid is a delta in the matching direction. Whenever the next cell holds a mirror (idealKind == "mirror"), the lookup table _mirror_redirect applies the 45° fold. The walk terminates at the first detector or after 256 empty cells (safety bound).

The mirror fold table

The 16-entry lookup is the heart of folded paths. Read it as (beam in, mirror yaw) → beam out:

beam in ↓ · mirror yaw →0°90°180°270°
+x+y−y−y+y
+y−x+x+x−x
−x−y+y+y−y
−y+x−x−x+x
Known limitation. The router is 2D today: it walks the (ix, iy) plane only. +z/−z propagation is in the axis table but not exercised by the source-yaw selector. Beamsplitters drop the transmitted arm — see PLAN §9 (multi-chain branching).

What a thin lens build looks like

builders.py · _ideal_thin_lens()def _ideal_thin_lens(p):
    f         = float(p.get("focalLength", 100.0)) or 100.0
    diameter  = float(p.get("diameter", 25.4))
    thickness = float(p.get("thickness", 2.5))
    n_d       = 1.5168                      # N-BK7 default
    radius_default = 2.0 * (n_d - 1.0) * abs(f)   # lens-maker, equiconvex
    sign      = 1.0 if f > 0 else -1.0
    r1, r2    = sign * radius_default, -sign * radius_default
    return {"surface_group": {"surfaces": [
        _surface(z_offset_mm=0,         radius=r1, material="N-BK7", is_stop=True, ...),
        _surface(z_offset_mm=thickness, radius=r2, material="air",  ...),
    ]}}

Source elements are special — they emit no surfaces but carry a wavelength_um field that _wavelengths_from_chain picks up downstream.

External .optiland.json or .zmx files are loaded via Optiland's own readers, then projected onto the offset-based dict shape. The object surface (index 0) and image surface (last) are stripped so the embedded sub-system splices cleanly into the running cube layout:

builders.py · _component_from_optic()def _component_from_optic(optic):
    surfaces_in = list(optic.surface_group.surfaces)
    inner       = surfaces_in[1:-1]               # drop object + image
    z0          = float(inner[0].geometry.cs.z)

    out = []
    for surf in inner:
        cs            = surf.geometry.cs
        radius        = float(getattr(surf.geometry, "radius", math.inf))
        is_reflective = bool(getattr(surf.interaction_model, "is_reflective", False))
        out.append(_surface(
            z_offset_mm = float(cs.z) - z0,    # relative to module anchor
            radius      = radius,
            material    = "mirror" if is_reflective else material_name,
            ...
        ))
    return {"surface_group": {"surfaces": out}, "module_length_mm": ...}

This is what gives lens-pos-50mm/optics/optiland.json primacy over the CSV: when present, the registry rewrites that module's spec to kind: optiland_json, the builder follows this path, and you get the real designed surfaces with their actual radii and coatings.

A vendor catalog is a folder containing index.json plus a stack of Zemax files. Each index entry has at minimum an sku and a focal_mm:

openUC2-optikit-modules/vendor-catalogs/thorlabs/index.json[
  {"sku": "LA1131-A", "zmx": "LA1131-A.zmx", "focal_mm": 50.0, "diameter_mm": 25.4},
  {"sku": "LA1050-A", "zmx": "LA1050-A.zmx", "focal_mm": 100.0, "diameter_mm": 25.4}
]

Resolution either pins the SKU explicitly or falls back to the closest focal-length match within ±5 %:

vendor_catalog.pyentry = (cat.by_sku(sku) if sku else
         cat.best_focal_match(float(params.get("focalLength", 100.0))))
if entry is None:
    return _ideal_thin_lens(params)        # graceful fallback
return _component_from_optic(cat.load_optic(entry))
Catalog files may be local paths or https URLs — Optiland's Zemax reader handles both, so vendor catalogs can live behind a static GitHub raw mirror without code changes.
07 Stage 6 · Surface insertion

Absolute z, cumulative anchor.

Iteration 2 dropped the old append-and-fix-up-the-thickness dance. The orchestrator now keeps a z_running counter and calls surfaces.add(z=…) with absolute coordinates.

The anchor formula

Each module's anchor is the cube centre, plus the per-module dx/dy/dz, with dz projected onto the incoming propagation axis. So a +5 mm dz means "5 mm further along the beam," not "5 mm in world z":

optikit_to_optiland.py · _element_anchor()cx, cy, cz = stop.element.cube_center_mm
dx, dy, dz = p["dx_mm"], p["dy_mm"], p["dz_mm"]
axis       = stop.prop_axis_in
sign       = +1 if axis.startswith("+") else -1

if axis.endswith("x"):    cx += sign*dz; cy += dy;       cz += dx
elif axis.endswith("y"):  cy += sign*dz; cx += dx;       cz += dy
else:                    cz += sign*dz; cx += dx;       cy += dy

The orchestrator loop

For every chain stop, the orchestrator computes the new anchor, projects the gap from the previous anchor onto the previous propagation axis, accumulates that gap into z_running, and then calls the builder. The result is one absolute z per surface:

optikit_to_optiland.py · _build_optic_for_chain()z_running = 0.0
for stop in chain.stops:
    spec      = registry.resolve(stop.element.module_id, stop.element.params)
    if spec["ideal_kind"] == "source": # sources contribute only wavelengths
        continue
    component = build_component_json(spec)
    surfaces  = component["surface_group"]["surfaces"]
    if not surfaces:                       # passive cube
        continue

    this_anchor      = _element_anchor(stop, spec)
    gap              = abs(_gap_along(prev_anchor, this_anchor, prev_axis))
    z_running       += gap
    module_anchor_z  = z_running

    _insert_surfaces(sys_op, module_anchor_z, spec, component)

    # advance to the last surface so the next gap stacks on top
    z_running = module_anchor_z + float(surfaces[-1]["z_offset_mm"])
    prev_anchor, prev_axis = this_anchor, stop.prop_axis_out

Inside _insert_surfaces

The inserter unpacks each surface dict, adds the per-module rotation offsets (CSV: rxOffset_deg / ryOffset_deg / rzOffset_deg) to per-surface rotations, and calls Optiland's surfaces.add with everything at once:

optikit_to_optiland.py · _insert_surfaces()for sj in surfaces:
    z_abs = module_anchor_z_mm + float(sj["z_offset_mm"])
    sys_op.surfaces.add(
        index   = sys_op.surface_group.num_surfaces,
        z       = z_abs,
        x       = sj["x_offset_mm"],
        y       = sj["y_offset_mm"],
        radius  = sj["geometry"]["radius"],
        conic   = sj["geometry"]["conic"],
        material= sj.get("material"),
        is_stop = sj.get("is_stop", False),
        rx      = math.radians(sj["rx_deg"]) + rx_off,
        ry      = math.radians(sj["ry_deg"]) + ry_off,
        rz      = math.radians(sj["rz_deg"]) + rz_off,
        aperture= sj.get("aperture_diameter"),
    )
Why absolute-z matters. The previous iteration appended each surface and then back-edited surfaces[-1].thickness to fix the gap. That made folded paths painful and forced bare try/except: pass around every gap fix-up. Switching to absolute z gives folded paths "for free" — a 90° mirror just sets rx = π/4 on its own surface; no parallel state machine.
08 Stage 7 · Output

One file per chain, plus a summary.

After every chain is built, the orchestrator writes one .optiland.json per chain plus a single .summary.json with paraxial properties.

Files written

  • <layout>.optiland.json — the primary chain (chain 1). Loadable in the Optiland GUI and via Optic.from_dict(json.load(f)).
  • <layout>.chain-N.optiland.json — one file per additional chain (chain 2, 3, …). One chain per detected light source.
  • <layout>.summary.json — paraxial summary across all chains.

The summary file

<layout>.summary.json{
  "layout":        "my-bench",
  "grid_pitch_mm": 50.0,
  "chains": [
    {
      "source":        "Source_00",
      "module_id":     "laser-635nm",
      "stops":         ["laser-635nm", "mirror-1x1", "lens-pos-1x1", "camera-usb"],
      "n_surfaces":    5,
      "n_wavelengths": 1,
      "efl_mm":        50.0
    }
  ]
}

If Optiland's Paraxial raises (e.g. underdetermined chain), the chain entry carries paraxial_error instead of efl_mm. The export itself never fails on these — it just records the diagnostic.
